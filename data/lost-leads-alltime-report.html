All-Time Investigation Report

56 Paid Leads Lost Over 3 Months

Between December 26, 2025 and March 26, 2026, the Mira Mar website contact form silently rejected 56 legitimate lead submissions. The problem existed for at least 3 months before it was discovered — every visitor who filled out the form thought they submitted it successfully, but Spark never received their data. This report covers the full scope of the issue, including 24 additional leads found beyond the original 32-person investigation.

Period: Dec 26, 2025 – Mar 26, 2026 Prepared: March 26, 2026 Source: CallRail × Spark × Failed Submissions Log
01 — Executive Summary

What Happened

The website contact form posted directly to Spark’s servers (cross-origin). When visitors arrived from social media apps (Facebook, Instagram) or certain browsers, Spark’s reCAPTCHA verification failed — silently blocking the submission. The visitor saw no error. CallRail captured the form data, but Spark never received it. This issue persisted from at least late December 2025 through March 24, 2026 when the fix was deployed.

02 — Root Cause Analysis

Why Spark Rejected These Leads

183 total form submissions were rejected by Spark over the investigation period. The 55 real leads in this report were blocked by reCAPTCHA-related failures. The rejection reasons break down into clear categories.

🔒

reCAPTCHA Token Missing

The form posted to Spark without a reCAPTCHA token. This happens when the browser blocks the reCAPTCHA script from loading — common in Facebook/Instagram in-app browsers that restrict third-party JavaScript.

05 — Resolution

How This Was Fixed

The root cause was that the website form posted directly to Spark’s servers (cross-origin). This caused browsers to block reCAPTCHA scripts, especially in social media in-app browsers. The fix was deployed on March 24, 2026.

All 55 leads have been recovered

Every affected contact was pushed into Spark via the API with their correct marketing source and UTM tracking data. The all-time investigation uncovered 24 additional leads beyond the original 32-person report — contacts lost as far back as December 26, 2025 that were never detected because the failed submissions log only captured failures from early March onward. These earlier leads were found by cross-referencing CallRail form submissions against Spark records.

Step 1
Server-Side Form Relay

Forms now POST to the WordPress server first, which relays to Spark server-side. This eliminates the cross-origin reCAPTCHA problem entirely.

Step 2
Lead Recovery via API

All 55 affected contacts were pushed to Spark via the API with their original marketing source and UTM campaign data preserved.

Step 3
Monitoring Deployed

UTM cookie diagnostics are now tracking every visitor to confirm the fix is working. The reconciliation dashboard provides ongoing monitoring.

Important Caveat: Spark Registration Dates

The Spark API does not allow backdating. Contacts recovered via the API push show a Spark registration date of March 24, 2026 (the date they were pushed), not their original form submission date. Some earlier contacts (Dec–Feb) may have already been in Spark from a successful retry — these show an earlier Spark date. The CallRail Date column in the table above always reflects the actual date each person submitted the form.

Scope Expansion: From 32 to 55 Leads

The original investigation (March 7–24) identified 32 affected leads using the failed submissions log that began recording in early March. Expanding the analysis to cover all time (December 26, 2025 – March 26, 2026) revealed 24 additional leads from December through early March. These earlier failures were discovered by cross-referencing every CallRail form submission against Spark contact records. The same reCAPTCHA bug was responsible — it simply went undetected because no error logging existed before March.
